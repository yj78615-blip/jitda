01개요 · 위협 모델

1.1기본 원칙

  1. 서버가 최종 방어선. 프론트엔드 검증은 UX 편의일 뿐, 모든 규칙은 서버에서 다시 강제한다.
  2. 기본은 거부(deny by default). 화이트리스트 방식 — 허용된 것만 통과.
  3. 최소 권한. 유저·서비스·DB 롤 모두 필요한 권한만.
  4. 비밀은 코드 밖. 환경변수 · 시크릿 매니저. 커밋 금지.
  5. 결제·개인정보는 우리가 저장하지 않는다. Stripe에 위임.

02전송 계층 · 헤더

2.1HTTPS 전면 강제

  • 모든 도메인(jitda.com, api.jitda.com, cdn.jitda.com)은 HTTPS 필수.
  • HTTP 요청은 CloudFlare/Vercel 엣지에서 301로 HTTPS 리다이렉트.
  • TLS 1.2 이상만. TLS 1.0 · 1.1 · SSLv3 비활성.
  • 인증서: Let's Encrypt (CloudFlare 자동) 또는 Vercel 관리형.

2.2HSTS · 필수 응답 헤더

모든 응답에 다음 헤더를 세팅한다 (프록시 또는 애플리케이션 미들웨어).

Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
Content-Security-Policy:  default-src 'self';
                          img-src 'self' https://cdn.jitda.com data:;
                          script-src 'self';
                          style-src 'self' 'unsafe-inline';
                          connect-src 'self' https://api.jitda.com https://api.stripe.com;
                          frame-src https://js.stripe.com https://hooks.stripe.com;
                          object-src 'none';
                          base-uri 'self';
                          form-action 'self';
                          frame-ancestors 'none';
                          upgrade-insecure-requests
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(self)
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Resource-Policy: same-site
CSP 예외: Stripe.js 로드를 위해 script-src에 https://js.stripe.com이 필요할 수 있음. 최소한으로만 열고, nonce·hash 방식이 가능하면 그쪽으로 이관.

2.3CORS

  • API 서버는 Access-Control-Allow-Origin을 화이트리스트로만 반환 (https://jitda.com, staging 도메인). 와일드카드 금지.
  • Access-Control-Allow-Credentials: true는 필요한 경로만.

3.2패스워드 해싱 — Argon2id

알고리즘: argon2id 강제. bcrypt는 레거시 마이그레이션 용도로만. 
// Node.js — @node-rs/argon2 또는 argon2 라이브러리
import { hash, verify, Algorithm } from '@node-rs/argon2';

const HASH_OPTS = {
  algorithm: Algorithm.Argon2id,
  memoryCost: 19_456,     // 19 MiB (OWASP 2024 권장)
  timeCost: 2,            // 2 iterations
  parallelism: 1,
  outputLen: 32,
  // salt는 라이브러리가 자동 생성 (16 bytes)
};

// 저장
const password_hash = await hash(raw_password, HASH_OPTS);
// User.password_hash 컬럼에만 저장. 원문은 절대 로그·응답에 포함하지 않음.

// 검증
const ok = await verify(user.password_hash, submitted_password);
  • 파라미터는 서버 CPU에서 500ms~1s가 걸리도록 튜닝 (Argon2id 저항성).
  • 파라미터 상향 시: 로그인 성공 후 needsRehash() 체크하고 새 파라미터로 재해싱.
  • 비교는 constant-time 라이브러리 함수만 사용. 직접 === 금지.

3.3세션 — JWT + Refresh

Access token
JWT (HS256 · 서버 시크릿) · 1시간 만료 · 상태 없음
Refresh token
불투명 토큰(랜덤 256bit) · DB 저장 · 30일 만료 · 1회용 회전(rotation)
저장 위치
Refresh: HttpOnly · Secure · SameSite=Strict 쿠키. Access: 메모리 (localStorage 금지)
Revoke
Logout · 패스워드 변경 · 이상 감지 시 refresh 즉시 무효화
localStorage에 토큰 금지. XSS 발생 시 즉시 유출된다. HttpOnly 쿠키가 유일하게 안전한 위치.

3.4이메일 인증

  • 가입 직후 이메일 발송 → 링크에 단회용 토큰 (24시간 만료).
  • 미인증 계정은 구독·후원 결제 불가. 감상·팔로우는 허용.
  • 이메일 변경 시 재인증 필수.

3.5OAuth (Google · Apple)

  • Authorization Code + PKCE 방식만. Implicit flow 금지.
  • state 파라미터로 CSRF 방어 (crypto random, 세션에 바인딩).
  • redirect_uri는 OAuth 콘솔에 등록된 정확한 값만.
  • 서버는 ID 토큰 서명을 provider의 jwks_uri로 검증 (kid · aud · iss · exp).
  • 이메일 미인증(email_verified: false) 계정은 거부.

3.6로그인 이상 감지 · 잠금

  • 같은 계정에 대한 로그인 실패 5회 / 15분 초과 시 15분 잠금.
  • 같은 IP에서 여러 계정 시도 50회 / 시간 초과 시 IP 차단.
  • 새 국가·기기에서 로그인 시 이메일 알림.
  • 차단은 토큰버킷 + 지수 백오프. 응답 시간 정보 누출 방지.

04인가 · 권한

4.1역할

role권한 요약
user본인 프로필·구독·팁 관리, 공개 콘텐츠 열람, 구독한 작가의 유료 회차 열람
admin+ 신고 처리 · 정산 실행 · 유저 상태 변경 · 전역 설정

모든 유저는 자기 프로필을 author로 활성화할 수 있음(오픈 플랫폼 원칙). "AUTHOR" 태그는 본인 자원에 한정되는 인가.

4.2리소스 소유권 검사

모든 mutating 엔드포인트(PATCH·DELETE·게시)는 요청자가 리소스 소유자인지 서버가 재검증한다.

// PATCH /series/{id}
const series = await db.series.findUnique({ where: { id }});
if (!series) return notFound();
if (series.author_id !== ctx.user.id) return forbidden();
// ... 이후 업데이트

4.3유료 콘텐츠 접근

  1. 클라이언트에서 잠금 UI를 표시하는 것은 UX 편의. 서버가 매번 확인.
  2. GET /episodes/{id} 진입 시 회차가 is_subscriber_only이면 활성 구독을 DB에서 조회.
  3. 비구독자 요청은 403 forbidden + details.required_subscription: true.
  4. 이미지 URL은 서명된 URL(단기 만료 15분)로 발급 — CDN에서도 접근 통제 가능.
안티패턴: 프론트에서 잠금만 걸고 실제 이미지는 공개 URL로 서빙하면, 개발자 도구로 URL을 뽑아 무단 열람할 수 있음. 서명 URL 또는 인증 프록시 필수.

05입력 검증

5.1계층 원칙

  1. 프론트 (UX): 즉각 피드백 · 잘못된 요청 방지 · 필드별 힌트
  2. API 게이트웨이 (형식): Content-Type · 크기 · 스키마
  3. 서버 핸들러 (스키마): Zod · Pydantic · io-ts 등으로 타입·범위·형식 검증
  4. 비즈니스 룰 (상태): 소유권 · 결제 상태 · 정산 가능 여부
  5. DB 스키마 (최종): NOT NULL · CHECK · FK 로 최후 방어

5.2스키마 예시 (Zod)

// POST /auth/signup
const SignupSchema = z.object({
  email: z.string().email().max(254).toLowerCase(),
  password: z.string().min(10).max(128)
    .refine((p) => zxcvbn(p).score >= 3, '너무 흔한 비밀번호'),
  display_name: z.string().min(1).max(30).trim()
    .refine((n) => !/[\p{Cc}\p{Cf}]/u.test(n), '제어 문자 포함 불가'),
  handle: z.string().min(3).max(20).regex(/^[a-z0-9_]+$/,
    '소문자·숫자·언더스코어만 (3–20자)')
});

// PATCH /authors/me
const AuthorPatchSchema = z.object({
  subscription: z.object({
    enabled: z.boolean().optional(),
    price: z.number().int().min(1000).max(30000).optional(),
    currency: z.enum(['KRW']).optional()
  }).optional()
});

// POST /tips
const TipSchema = z.object({
  author_id: z.string().regex(/^usr_[a-zA-Z0-9]{4,}$/),
  amount: z.number().int().min(1000).max(1_000_000),
  currency: z.enum(['KRW', 'USD']),
  message: z.string().max(140).optional(),
  message_public: z.boolean().optional()
});

5.3정규화

  • 이메일: 소문자 · 앞뒤 공백 제거 · + 뒷부분(예: a+spam@x.com)은 그대로 저장하되 조회 인덱스는 정규화된 것으로도 유지.
  • Handle: 소문자만. 유니코드 confusables 정규화 (NFKC) 후 ASCII 강제.
  • 텍스트 필드: NFC 정규화. Zero-width joiner/space 제거.

06XSS 방어

6.1기본 원칙

  1. 모든 사용자 입력은 렌더 시점에 이스케이프.
  2. React·Vue·Svelte 등 프레임워크의 자동 이스케이프에 의존. innerHTML·dangerouslySetInnerHTML·v-html은 금지. 예외는 코드 리뷰 게이팅.
  3. 서버 응답 JSON을 HTML 컨텍스트에 직접 삽입 금지 (템플릿 리터럴로 ${data} 넣기 등).
  4. CSP로 인라인 스크립트 차단 → 침해되어도 스크립트 실행 저지.

6.2사용자 HTML (bio · 시리즈 description)

마크다운 일부를 허용할 경우 DOMPurify + allowlist. 인라인 스타일 · 스크립트 · 이벤트 핸들러 전부 제거.

import DOMPurify from 'isomorphic-dompurify';

const clean = DOMPurify.sanitize(userMarkdownRenderedHTML, {
  ALLOWED_TAGS: ['p', 'br', 'strong', 'em', 'a', 'ul', 'ol', 'li', 'blockquote', 'code'],
  ALLOWED_ATTR: ['href'],
  ALLOWED_URI_REGEXP: /^(https?|mailto):/i,
  // rel="noopener noreferrer" 강제
  ADD_ATTR: ['target'],
});
  • 모든 <a>는 rel="noopener noreferrer" · target="_blank" 강제.
  • 이미지 · 스크립트 · iframe 태그 전면 금지.

6.3URL 검증

외부 링크(작가 프로필의 external_links 등)는 스킴 화이트리스트: http:, https:, mailto:만. javascript:, data:, vbscript: 거부.

6.4현재 목업의 XSS 방어 감사

확인: 현재까지 만든 4개 목업(메인·작가·뷰어·스튜디오)은 모두 escapeHtml()을 렌더 시 사용. 유저 입력이 HTML 컨텍스트에 들어가는 지점을 감사했음. 
// 목업의 escapeHtml 구현 (참고)
const escapeHtml = (s) => String(s ?? '').replace(/[&<>"']/g, (c) => ({
  '&': '&amp;', '<': '&lt;', '>': '&gt;',
  '"': '&quot;', "'": '&#39;'
}[c]));

07Injection 방어

7.1SQL · ORM

  • Prisma · Drizzle · Kysely 등 파라미터 바인딩이 강제되는 ORM만 사용.
  • 원시 SQL($queryRaw·db.raw)은 리뷰 게이팅. 사용해야 하면 placeholder 필수.
// ❌ 절대 금지
db.$queryRawUnsafe(`SELECT * FROM users WHERE email = '${email}'`);

// ✅ 파라미터 바인딩
db.$queryRaw`SELECT * FROM users WHERE email = ${email}`;

// ✅ ORM 쿼리 빌더
db.user.findUnique({ where: { email } });

7.2NoSQL Injection

사용자가 보낸 값이 쿼리 오브젝트로 바로 들어가는 것 금지. 항상 스키마 검증 후 사용.

// ❌ Mongo에서 { email: { $ne: null } } 로 조작 가능
db.users.findOne({ email: req.body.email });

// ✅ 스키마 검증 후 문자열만 허용
const { email } = SignupSchema.parse(req.body);
db.users.findOne({ email });

7.3Command · Path Injection

  • 서버가 이미지 처리(sharp·ffmpeg) 시 파일명 · 경로에 사용자 입력 사용 금지. 항상 UUID/자체 생성 ID로 치환.
  • Shell exec는 사용하지 않는다. 라이브러리 API만.
  • 사용자 제공 URL을 서버가 fetch하는 경우 SSRF 방어: DNS 해석 후 사설 IP 대역(10.x·192.168.x·127.x·169.254.x·::1) 차단.

7.4SSRF · 헤더 인젝션

  • fetch 시 redirect follow 제한 (예: max 3) — 리다이렉트로 내부망 우회 방지.
  • 사용자 입력을 응답 헤더에 넣지 말 것 (CRLF 인젝션). 넣어야 하면 \r\n 제거.

08CSRF 방어

  1. Refresh token 쿠키: SameSite=Strict; HttpOnly; Secure — 크로스 오리진 요청에 자동 전송 안 됨.
  2. Access token: Authorization: Bearer 헤더로만 전송 — 브라우저가 자동으로 붙이지 않으므로 CSRF 자체가 성립 안 함.
  3. 쿠키 세션을 병행한다면 double submit cookie 또는 synchronizer token. 두 접근을 섞지 말 것.
  4. 상태 변경(POST·PATCH·DELETE) 시 Origin·Referer 헤더가 화이트리스트인지 확인 (defense in depth).
  5. Stripe 웹훅 엔드포인트는 CSRF 미들웨어 bypass 필요 (서명 검증으로 대체).

9.1조회수 어뷰징 방어

  • 같은 (episode, user or session)에 대한 조회는 1시간에 1회만 카운트.
  • 세션 없는 익명 조회: IP + User-Agent 지문으로 중복 필터.
  • 신규 계정(가입 후 7일 미만)의 조회·좋아요·구독 가중치는 0.5배.
  • 비정상 패턴(초당 100 조회 등)은 자동 flag → 관리자 큐.

10업로드 보안

10.1presigned URL 방식

서버가 다운로드/업로드 URL을 직접 노출하지 않고 단기 만료(15분) presigned URL을 발급. 서버 대역폭 절약 + 접근 통제 유연성.

10.3이미지 서빙

  • 업로드된 파일은 원본 파일명을 노출하지 않고 내부 ID로만 접근.
  • CDN 응답에 Content-Type을 서버가 강제 세팅 (X-Content-Type-Options: nosniff 필수).
  • 구독자 전용 이미지는 서명된 URL (15분 만료) — CDN 캐시 키에 서명 미포함으로 캐싱은 유지.

11결제 · Webhook

11.1PCI DSS 회피

  • 카드 정보는 Stripe Elements iframe에서만 입력받는다. 우리 서버는 절대 카드 원문을 받지 않는다.
  • PCI DSS SAQ A 범주(카드 데이터 처리 최소).

11.2결제 흐름

  1. 클라이언트: POST /subscriptions or /tips → 서버가 Stripe PaymentIntent 생성 → client_secret 반환
  2. 클라이언트: Stripe.js 로 결제 수행
  3. Stripe → 우리 서버 Webhook: payment_intent.succeeded 수신 → DB 반영
  4. DB 반영은 Webhook 이 유일한 진실. 클라이언트 콜백만 믿고 활성화하지 않는다.

11.3Webhook 서명 검증

import Stripe from 'stripe';
const stripe = new Stripe(SECRET);

// Express 예시
app.post('/api/v1/webhooks/stripe',
  express.raw({ type: 'application/json' }),  // 원문 필수!
  async (req, res) => {
    const sig = req.headers['stripe-signature'];
    let event;
    try {
      event = stripe.webhooks.constructEvent(req.body, sig, WEBHOOK_SECRET);
    } catch (err) {
      return res.status(400).send('signature verification failed');
    }
    // Idempotency: event.id 를 DB 에 기록. 중복 수신 방지.
    if (await eventAlreadyProcessed(event.id)) return res.json({ received: true });
    await handleEvent(event);
    await markEventProcessed(event.id);
    res.json({ received: true });
});

11.4정산 · 이체

  • 정산 배치는 periodic idempotency key({author_id, period})로 중복 지급 방지.
  • Payout 관련 액션은 반드시 감사 로그 기록.
  • 관리자 수동 트리거는 2FA 필수 (v2).

12PII · 데이터 보호

12.1수집 최소화

  • 필수: 이메일, 패스워드 해시, display_name, handle.
  • 선택: bio, 아바타, 외부 링크. 실명 · 주소 · 전화 수집 안 함.
  • 연령 인증(Phase 2)은 인증만 확인하고 생년월일 자체는 저장하지 않는다.
  • 정산용 계좌 정보는 우리가 저장 안 함 (Stripe Connect).

12.2암호화

전송

12.3탈퇴 · 삭제

  • 탈퇴 요청 시 30일 유예. 유예 중 로그인 시 복구 가능.
  • 유예 후: PII를 익명화 (이메일 · 닉네임 마스킹, 회차·댓글은 "탈퇴한 사용자" 표시로 유지).
  • 사용자 요청 시 개인정보 다운로드 (GDPR-style export).

12.4제3자 공유

  • Stripe(결제) · Cloudflare(CDN) · Resend(이메일) · Sentry(오류) · PostHog(분석) — 목록을 개인정보처리방침에 명시.
  • Sentry에는 요청 body를 기본 스크러빙. 이메일·토큰·body 전체 자동 마스킹.
  • PostHog는 autocapture 비활성. 이벤트를 명시적으로 지정.

13.3보관 · 접근

14비밀 · 의존성

14.1시크릿

  • 환경 변수만 사용. 코드에 하드코드 금지 (린트로 강제 · pre-commit 훅).
  • Vercel · Fly · AWS Secrets Manager 등 관리형 시크릿.
  • DB · Stripe · JWT · Webhook 시크릿은 환경별로 분리 (dev · staging · prod).
  • 키 로테이션: JWT signing key는 90일마다. 이전 키로 발급된 토큰은 만료까지 유효.
  • git leak 스캐너: gitleaks 를 CI에 포함.

14.2의존성

  • npm audit · Renovate 봇 · GitHub Dependabot 활성.
  • Critical CVE는 24시간 내 패치. High는 1주.
  • 새 dependency 추가 시 리뷰. 다운로드 수·유지보수·라이선스 확인.
  • Lock 파일 커밋 필수. npm ci 로만 배포.

15배포 체크리스트

Phase 1 프로덕션 배포 전 반드시 확인.

  • 모든 도메인 HTTPS 강제 (HTTP → 301 리다이렉트)
  • HSTS 헤더 (max-age ≥ 1년, includeSubDomains, preload)
  • CSP 헤더 설정 · 인라인 스크립트 제거
  • 보안 응답 헤더(X-Content-Type-Options, Referrer-Policy 등) 전부
  • CORS 화이트리스트만 허용, wildcard 없음
  • Argon2id 파라미터 튜닝 (단위 테스트로 목표 시간 검증)
  • zxcvbn · haveibeenpwned 연동
  • Refresh token: HttpOnly + Secure + SameSite=Strict
  • Access token: localStorage 절대 미사용
  • 이메일 인증 흐름 · 재전송 레이트 리밋
  • OAuth: PKCE · state 검증 · ID 토큰 JWKS 검증
  • 로그인 5회 실패 잠금 · 새 국가 로그인 알림
  • 모든 mutating 엔드포인트에서 리소스 소유권 재확인
  • Zod 스키마로 전 엔드포인트 입력 검증
  • 사용자 입력 렌더 시 이스케이프 · DOMPurify 적용 지점 리뷰
  • URL 스킴 화이트리스트 검증
  • ORM 파라미터 바인딩만 사용 · $queryRawUnsafe grep 확인
  • SSRF 방어 (사설 IP 대역 차단, redirect 제한)
  • 레이트 리밋 미들웨어 활성 · 429 응답 확인
  • 업로드: MIME magic bytes · 재인코딩 · EXIF 제거
  • 구독자 전용 이미지 서명 URL
  • Stripe Webhook 서명 검증 · idempotency
  • 정산 배치 idempotency key
  • 감사 로그 append-only DB 계정 분리
  • Sentry · PostHog PII 스크러빙 설정
  • gitleaks · npm audit CI 통과
  • 비밀 환경별 분리 · git 히스토리에 커밋된 비밀 없음
  • 탈퇴 · export 흐름 QA
  • 침투 테스트 (최소 자동 스캐너: zap-baseline) 통과
