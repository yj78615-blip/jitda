웹툰 작가 오픈 플랫폼

누구나 자신의 웹툰·일러스트를 올리고, 팬을 모으고, 유료 구독·후원으로 수익화할 수 있는 오픈 플랫폼.

01제품 개요

1.1배경

  • 국내 웹툰 플랫폼(네이버·카카오)은 진입 심사가 높아 신인·아마추어의 발표 장벽이 크다.
  • 포스타입·픽시브·Patreon은 각각 한계 존재 — 수익모델 미비, 지역 편향, 웹툰 최적화 부재.
  • "오픈 게시 + 팬 기반 수익화 + 웹툰 최적 뷰어"를 한국·영어권 동시 타겟으로 잡을 여지가 있다.

1.2제품 목표

  1. 작가가 5분 안에 계정을 만들고 첫 작품을 올릴 수 있다.
  2. 독자가 로그인 없이도 랭킹·장르 카테고리로 원하는 작품을 발견할 수 있다.
  3. 팬은 무료(팔로우)와 유료(구독·후원) 두 층위로 작가를 지원할 수 있다.
  4. 작가는 월 1회 정산으로 실수령금을 자동으로 받는다.

1.3성공 지표 (KPI)

층위지표
유입MAU · 회차 조회수 · 검색 노출 클릭율
인게이지먼트팔로우 전환율 · D7/D30 재방문율 · 회차당 평균 체류시간
수익화유료 구독 전환율(팔로워→구독자) · ARPPU · 후원 건수
공급신규 작가 등록 수 · 주간 신규 회차 수 · 활성 작가 비율

02타겟 사용자

2.1작가 (공급자)

  • 아마추어~세미프로 웹툰·일러스트 작가
  • 자기 페이지를 갖고 싶고, 팬 기반의 정기 수익이 필요함
  • 한국어·영어 독자를 함께 만나고 싶음

2.2독자 (수요자)

  • 웹툰·일러스트를 즐기는 10대 후반 ~ 30대
  • 좋아하는 작가에게 소액이라도 직접 지불할 의향 있음
  • 데스크톱·모바일 웹 브라우저로 접근

2.3관리자 (내부)

  • 신고 처리, 정산, 저작권 침해 대응, 성인 콘텐츠 심의(Phase 2)

03핵심 사용자 시나리오

  1. 작가 온보딩 — 소셜·이메일 가입 → 프로필 세팅 → 구독료 설정(선택) → 첫 회차 업로드
  2. 독자 발견 — 메인 접속 → "인기 작품 TOP 10" 탭 확인 → 작품 상세 진입 → 무료 회차 감상 → 마음에 들면 팔로우
  3. 팬 전환 — 팔로우 상태에서 "구독자 전용" 회차 배지 노출 → 구독 결제 → 잠금 해제
  4. 일회성 후원 — 회차 감상 후 하단 "팁" 버튼 → 자유 금액 입력 → 결제
  5. 작가 정산 — 매월 말 자동 집계 → 이메일로 정산 명세 → 익월 중순 계좌 입금

4.2메인 화면 구성

  • 상단: 로고 · 검색바 · 로그인·가입 or 프로필 메뉴
  • 탭 히어로 영역: 인기 작품 TOP 10 · 인기 포스트 TOP 10 · 최신작 TOP 3

  • 최근 팔로우 작가의 새 회차 (로그인 시)

05기능 요구사항

5.1계정 · 인증

  • 가입/로그인: 이메일+비밀번호 · Google OAuth · Apple OAuth
  • 프로필 필드: 필명, 자기소개(마크다운 일부), 아바타, 배너, 외부 링크
  • 작가 활성화: 모든 유저는 기본으로 "작가"가 될 수 있다 (별도 신청 불필요, 오픈 플랫폼 원칙)
  • 연령 확인 (Phase 2): 성인 콘텐츠 접근 시 별도 인증 (본인인증·신용카드 인증 등)
  • 탈퇴: 30일 유예 후 계정 삭제, 유료 구독 자동 해지

5.2작가 페이지 (/@{author})

  • 프로필 헤더: 배너 · 아바타 · 자기소개 · 팔로워 수 · 구독자 수
  • 버튼: 팔로우 (무료) · 구독하기 (월 ₩X,XXX) · 팁 보내기
  • 탭: 시리즈 · 개별 포스트 · 사랑방(구독자 전용, Phase 2)
  • 정렬: 최신순 · 인기순

5.3작품 · 회차 관리

콘텐츠 단위

  • 시리즈(작품): 여러 회차를 묶는 단위 (연재물)
  • 개별 포스트: 단독 게시물 (일상 그림·낙서·단편)

회차·포스트 필드

  • 제목 · 설명 · 태그(장르 카테고리 중 선택 + 자유 태그 몇 개)
  • 이미지 파일들 (드래그앤드롭, 순서 조정)
  • 뷰어 형식: 세로 스크롤 · 페이지 넘기기 중 선택
  • 공개 범위: 전체 공개 · 구독자 전용
  • 예약 발행 시간
  • 성인 콘텐츠 여부 (Phase 2)

업로드 제약: 최대 파일 크기 회당 100MB 예상. 이미지 자동 리사이즈·WebP 변환은 백엔드에서 처리.

5.4뷰어

  • 세로 스크롤 모드: 이미지들을 세로로 이어붙여 렌더링 (한국 웹툰 표준)
  • 페이지 모드: 좌·우 페이지 넘기기, 키보드 화살표 지원
  • 공통 하단 액션: 이전 회차 / 다음 회차 · 좋아요 · 공유 · 팁 보내기 · 댓글
  • 구독자 전용 회차는 잠금 상태 UI + 구독하기 CTA

5.5발견 (탐색)

  • 장르 카테고리 페이지: 카테고리별 인기·최신 리스트 (/explore/romance 등)
  • 검색: 제목·태그·작가명 통합. Phase 1은 단순 LIKE 검색, Phase 2에서 검색 엔진 도입 검토.

5.6랭킹

  • 집계 대상: 시리즈(작품) · 개별 포스트 · 작가 3종
  • 인기 점수 공식 (초안): 
    score = w1·조회수(7d) + w2·좋아요(7d) + w3·댓글(7d)
      + w4·신규 팔로워(7d) + w5·신규 구독자(7d)
가중치는 관리자 콘솔에서 튜닝 가능한 변수로 설계
  • 어뷰징 방지: 동일 IP·계정 중복 조회 필터, 팔로워 없는 신규 계정 가중치 감소
  • 갱신 주기: 1시간마다 배치 재계산 (Phase 1). 실시간은 v2.

5.7팔로우 · 구독 · 후원

팔로우 (무료)

  • 원클릭. 로그인 시 새 회차 알림 자동 구독
  • 상호 팔로우 개념 없음(단방향)

구독 (유료 정기결제)

  • 작가가 단일 월 구독료 설정 (예: ₩3,000 ~ ₩30,000 사이, 관리자 캡 있음)
  • 결제 주기: 매월 결제일 = 최초 결제일 기준
  • 결제 실패 시 3회 재시도 후 자동 해지
  • 해지: 언제든 가능. 남은 기간까지는 접근 유지.

후원 (일회성 팁)

  • 자유 금액 or 프리셋 (₩1,000 · ₩3,000 · ₩5,000 · ₩10,000)
  • 짧은 메시지 첨부 가능 (공개·비공개 선택)
취소·환불 — 구독은 결제 후 7일 이내 미시청 시 환불 가능, 팁은 원칙적으로 환불 불가(부정 결제 예외).

5.8커뮤니티

  • 회차·포스트 댓글: 대댓글 1단계, 이모지 반응·좋아요, 작가의 고정·답변 표시, 삭제·신고
  • 구독자 전용 사랑방 (Phase 2): 작가 페이지 내 별도 게시판. 구독자만 글 작성.

5.9알림

  • 채널: 사이트 내 알림(종 아이콘) + 이메일
  • 트리거: 팔로우 중인 작가의 새 회차·포스트 · 내 회차·댓글에 댓글·좋아요·팁 · 새 구독자·팔로워 · 결제 성공·실패, 정산 완료
  • 설정: 종류별 on/off, 이메일 다이제스트(즉시·일간·주간)

5.10정산

  • 주기: 매월 1일 자동 집계 (전월 1일 ~ 말일 매출)
  • 차감: 결제 수수료(Stripe 실비) → 플랫폼 수수료(5~10%, 관리자 변수) → 환불 준비금
  • 최소 지급액: 입금 수수료를 상쇄하는 수준 (예: ₩10,000)
  • 지급일: 익월 15일 (은행 이체 or Stripe Connect Payout)
  • 작가 화면: 이번 달 예상 수익 · 이전 정산 내역 · 세금 관련 문서(간이 명세)

5.11모더레이션 · 저작권

신고 시스템 (사후 신고 기반)

  • 신고 카테고리: 저작권 침해 · 성인 콘텐츠(SFW 정책 위반) · 폭력·혐오 · 스팸 · 기타
  • 처리 SLA 목표: 72시간 이내 1차 응답
  • 위반 시: 콘텐츠 비공개 → 작가에게 통보·이의제기 창구
  • 누적 위반: 3회 이상 시 계정 정지 (관리자 재량)
  • 저작권 신고 폼: DMCA 유사 표준 (신고자 신원, 원본 링크, 침해 링크, 서약)

5.12관리자 콘솔

  • 신고 대기열, 처리 상태 관리
  • 정산 실행·검토
  • 작가·유저 검색 및 상태 변경
  • 랭킹 가중치 · 수수료율 · 구독료 캡 등 변수 조정
  • 통계 대시보드

07기술 요구사항

7.1프론트엔드

  • 프레임워크: Next.js (App Router, TypeScript) — SEO·이미지 최적화 필수
  • UI: TailwindCSS + shadcn/ui 또는 유사 컴포넌트 라이브러리
  • 상태 관리: React Query(서버 상태) + Zustand(로컬 상태) 정도로 최소화
  • i18n: next-intl 등으로 한/영 라우팅 분리 (/ko/, /en/)
  • 이미지: next/image, 뷰어에서는 lazy loading + prefetch 조합

7.2백엔드 (추후 구현)

처음부터 프론트/백 분리 전제 → 프론트는 REST/tRPC/GraphQL API를 소비.

후보 스택

  • A. Next.js API Routes + Prisma + PostgreSQL — 프론트와 동일 저장소, 초기 속도 최우선
  • B. NestJS + Prisma + PostgreSQL — 백엔드 팀이 별도로 붙을 때 확장성 확보
  • C. Django + DRF — 관리자·모더레이션 UI를 Django Admin에 얹기 가능
백엔드 통합 지점을 미리 설계 — 인증·저장·결제 각각의 이후 통합 지점을 프론트 코드에서 인터페이스 형태로 명확히 정의해 두어, 백엔드 구현 시 어댑터만 갈아끼움. 
interface AuthProvider { signIn(...): Promise<Session>; ... }
interface StorageProvider { uploadImage(file): Promise<ImageRef>; ... }
interface PaymentProvider { createSubscription(...): ...; createTip(...): ...; }

초기 개발 시에는 모두 로컬 목(mock) 구현으로 시작 → Phase 1 후반에 실제 서비스로 교체.

7.4인프라

  • DB: PostgreSQL (관리형 — Supabase · Neon · AWS RDS)
  • 배포: Vercel (프론트) + 별도 백엔드 서버(백엔드 도입 시)
  • 환경 분리: dev · staging · prod

08비기능 요구사항

  • 성능: 메인 페이지 LCP < 2.5s(모바일) · 뷰어 첫 이미지 표시 < 1.5s · API p95 < 300ms
  • 접근성: WCAG 2.1 AA 지향 (뷰어 키보드 조작, 스크린리더 alt 필수)
  • SEO: 작품·회차 페이지 서버 렌더링, OG 태그, sitemap.xml
  • 보안: HTTPS 강제, XSS·CSRF 대응, 결제 정보는 Stripe에 직접(자체 저장 금지)
  • 로그·감사: 결제·정산·모더레이션 액션은 감사 로그 필수

9.1콘텐츠 정책 (Phase 1)

  • Stripe 정책 검토 필수 — 필요 시 성인 전용 결제사(CCBill 등) 분리

9.3환불

  • 구독: 최초 결제 후 7일 이내, 회차 미열람 시 환불
  • 팁: 원칙적으로 환불 불가, 부정 결제 확인 시 예외

9.4저작권

  • 신고 폼 접수 후 72시간 내 1차 검토
  • 반복 위반자 3-strike 정책
