## 잘된 점
- 사용자 지적 (`@핸들` 이 자동차 핸들과 헷갈림) 을 즉시 수용 · 국내 관례 (`@아이디`) 로 정정. 브랜딩 관성보다 실제 사용자 이해를 우선.
- 진단 확장 (fields 노출) → 회원가입 실패 원인이 두루뭉술 "요청 값이 올바르지 않다" 에서 "handle: 3자 이상" 같이 구체화. 사용자 self-serve 가능.
- 배포 검증에 client 청크 grep 활용 — CSR 페이지도 배포 시점 확인 가능한 재사용 가능한 진단 레시피.

## 아쉬운 점

- Handle label 을 이번 세션 안에서 3번 변경 (URL 프로필 주소 → @핸들 → @아이디). 처음부터 국내 관례 조사 후 결정했으면 두 번 재작업 없음.
- 이전 회고들의 잔무 여전히 0건 처리: REFLECTION.md · 죽은 코드 · 스모크 테스트 · orphan 이미지 배치 · nonce-based CSP · seed 회차 추가.

## 배운 것
- 한국어 UI 용어는 영어 관례를 직역하지 말 것. `handle` → `핸들` (자동차) 오독. `아이디` 가 국내 표준.
- CSR (`'use client'`) 페이지 배포 검증은 HTML grep 무효 — Next.js 가 emit 하는 `page-{hash}.js` 청크를 직접 fetch 해서 확인해야.
- Zod validation error 를 클라에서 필드별로 풀어주면 사용자 self-serve 폭이 커짐. `withErrors` `internal_error` fallback 노출 (이전 세션) 과 같은 패턴.
